\section{Introducción}

El presente informe documenta el despliegue de una aplicación web dinámica basada en PHP y MariaDB, ejecutada mediante contenedores Docker y orquestada con Docker Compose. El objetivo del laboratorio fue implementar un stack web reproducible que permitiera separar la lógica de aplicación, la base de datos, la red de comunicación y la persistencia de información.

La aplicación desarrollada permite realizar operaciones CRUD: crear, leer, actualizar y eliminar registros desde una interfaz web. Para cumplir con los requisitos técnicos, se definió un contenedor de aplicación construido desde un \texttt{Dockerfile}, un contenedor de base de datos basado en una imagen oficial de MariaDB, un volumen para conservar datos y una red virtual de Docker para aislar la comunicación entre servicios.

\section{Desarrollo}

\subsection{Arquitectura de la solución}

La solución se organizó en una estructura de archivos compatible con la rúbrica del laboratorio. El servicio web se encuentra en la carpeta \texttt{app/}, la inicialización de la base de datos en \texttt{db/}, y la orquestación general en el archivo \texttt{docker-compose.yml}.

\subsection{Construcción}

La construcción de la imagen de la aplicación se realizó con el siguiente comando:

\begin{verbatim}
docker compose build
\end{verbatim}

Este comando lee el \texttt{Dockerfile} ubicado en la carpeta \texttt{app/}. Durante el proceso se descarga la imagen base \texttt{php:8.3-apache}, se instala la extensión \texttt{pdo\_mysql} y se copia el archivo \texttt{index.php} al directorio de publicación de Apache.

\subsection{Ejecución}

Una vez construida la imagen, los servicios se levantaron en segundo plano mediante:

\begin{verbatim}

\end{verbatim}

El parámetro \texttt{-d} permite ejecutar los contenedores en modo desacoplado. Luego se verificó el estado de los contenedores con:

\begin{verbatim}
docker ps
\end{verbatim}

En esta etapa se debe observar que los contenedores \texttt{laboratorio\_app} y \texttt{laboratorio\_db} se encuentren activos. Además, el puerto \texttt{8080} del equipo anfitrión queda asociado al puerto \texttt{80} del contenedor de aplicación.

\section{Configuración Docker}

\subsection{Dockerfile}

El archivo \texttt{app/Dockerfile} define la imagen de la aplicación:

\begin{verbatim}
FROM php:8.3-apache

RUN docker-php-ext-install pdo pdo_mysql

WORKDIR /var/www/html
COPY index.php /var/www/html/index.php

EXPOSE 80
\end{verbatim}

Esta configuración utiliza una imagen oficial de PHP con Apache, instala la extensión necesaria para conectarse a MySQL/MariaDB y copia el código fuente de la aplicación al directorio servido por Apache.

\subsection{Docker Compose}

El archivo \texttt{docker-compose.yml} define dos servicios principales. El servicio \texttt{app} construye la aplicación y publica el puerto \texttt{8080}; el servicio \texttt{db} levanta MariaDB, carga variables de entorno y usa un volumen persistente. También se incluye un \textit{healthcheck} para esperar a que la base de datos esté disponible antes de iniciar completamente el servicio web.

\subsection{Persistencia y aislamiento}

La persistencia se implementa mediante el volumen \texttt{db\_data}, montado en \texttt{/var/lib/mysql}. Esto evita que la información se pierda al detener o recrear los contenedores. El aislamiento se logra mediante la red \texttt{laboratorio\_net}, donde los servicios se comunican usando nombres internos definidos por Docker Compose.

\section{Conclusión}

La implementación del laboratorio permitió integrar una aplicación web dinámica en PHP con una base de datos MariaDB mediante Docker Compose. La separación entre el contenedor de aplicación y el contenedor de base de datos favorece una arquitectura más ordenada, reproducible y fácil de administrar.

El uso de un volumen persistente permitió conservar los registros creados desde la interfaz web, incluso después de reiniciar los contenedores. Asimismo, la red virtual de Docker permitió aislar la comunicación interna entre servicios, evitando exponer innecesariamente la base de datos.

En términos funcionales, la aplicación cumple con el objetivo principal del laboratorio al permitir crear, leer, actualizar y eliminar registros. Como mejoras futuras, se podrían incorporar autenticación de usuarios, validaciones más robustas, pruebas automatizadas y un diseño visual más avanzado para la interfaz.
